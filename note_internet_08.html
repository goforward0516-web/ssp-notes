第1章

システムの安定稼働と信頼性

なぜシステムの安定稼働が必要か

現代社会では多様なシステムが稼働しており、多くの人々がそれらに依存したサービスを利用している。← S03:p.6

重要

システムの停止はBCP（Business Continuity Plan：事業継続計画）に大きな影響を与える。企業の業務だけでなく、広範囲に社会的影響を及ぼす可能性もある。病院システムの停止は命に関わる問題に発展しうる。← S03:p.8

カテゴリ具体例
経済的損失ECサイト・決済サービス停止による収益損失 / 製造ラインの遅滞・停止による売上減少・生産量低下 / オンライン予約停止による機会損失
社会的影響交通システム停止で乗客への不利益・社会的混乱 / 通信サービス停止で機会損失・社会的混乱 / 医療システム停止で診療・手術中止、最悪の場合患者の命に関わる
信用の失墜不信感による顧客の離脱→収益減少 / 障害に伴う情報漏洩→個人情報・クレジットカード情報漏洩→損害賠償

予防保全の内容詳細

第2章

システムの仮想化技術

仮想化技術の概要とメリット

定義

仮想化技術とは、物理的なリソースを仮想的に分割・共有する技術。これにより、システムの柔軟性・可用性・負荷分散・省エネルギー化をもたらすことができる。← S03:p.27

メリット説明
ハードウェアの効率的な利用1つの物理サーバ上に複数の仮想サーバを構築→運用コスト削減。性能の余りを解消
柔軟性の向上仮想サーバの構築・移行が容易。複数アプリを同一HWで実行可能。環境構築・運用管理を簡略化
可用性の向上仮想サーバのホスト間移動が可能→サービス停止時間を最小化。バックアップ仮想サーバを複数用意して高可用化を実現
セキュリティの向上仮想マシンごとに独立したセキュリティレベルを設定可能。同一物理サーバ上でも異なるセキュリティポリシーを適用できる
省エネルギー化ハードウェアの効率的利用により電力消費を削減
← S03:p.28-29

サーバ仮想化（3種類）

1台の物理サーバ上に複数の仮想サーバを構築・運用する技術。ホストマシン上のOSや制御の仕組みに関わらず、仮想マシンごとに異なるOSを実行可能。← S03:p.32

代表的な実装例： VMware vSphere／Microsoft Hyper-V（Windowsの一部エディションに標準搭載）／Oracle VirtualBox← S03:p.32

外部補足・確認済: VMware公式・Microsoft公式

主要製品の現状（2024年時点）：
VMware vSphereはハイパーバイザ型の業界標準製品。2023年にBroadcomがVMwareを買収し、製品ラインナップや価格体系が変更されつつある。
Microsoft Hyper-VはWindows Server 2008以降に標準搭載（一部エディション）。追加コストなしで利用できるため中小規模のシステムで広く使われる。
Oracle VirtualBoxは無償利用可能なホストOS型の仮想化ソフトウェア。個人・開発用途に広く使われる。← S03:p.32

ストレージ仮想化

手法内容
ボリューム仮想化複数の物理ストレージを1つの仮想ストレージとして管理。RAID（Redundant Array of Independent Disks）が代表的
ブロックレベル仮想化物理ストレージをブロックとして管理し単一のプールとしてまとめる。サーバと物理ストレージの間に仮想化専用装置を配置。物理ストレージ上のデータ位置にとらわれず、サーバからは専用装置が示す仮想的なデータ配置が見える
ファイルレベル仮想化複数のサーバが同一の仮想的なファイルシステムを共有してアクセスする
← S03:p.34
外部補足・確認済: 各社公式情報

代表製品の現状：
NetAppはストレージ専業の独立系大手として現在も事業継続中。ONTAPをはじめとするデータ管理ソフトウェアで知られる。
EMC → Dell EMC → Dell Technologies：EMCは2016年9月にDellが約670億ドルで買収（当時IT業界史上最大）し「Dell EMC」に。2019年頃から「Dell Technologies」ブランドに移行。スライドの「EMC」は旧社名だが現在はDell Technologiesのストレージ部門として存続。← S03:p.30

ネットワーク仮想化

手法概要
VLAN（Virtual LAN）1つの物理ネットワーク上に複数の仮想ネットワークを構築・管理。セグメントを分離しトラフィックの分離とセキュリティ向上が可能。例: 一般ユーザー用NWと管理者用NWを分けて設定でき、相互参照できない
VPN（Virtual Private Network）パブリックネットワーク上にプライベートネットワークを構築。理論上パブリックNWと異なるNWを使うことで安全性が向上
SDN（Software Defined Network）ネットワークをプログラムで構築・制御可能にする技術。物理的なネットワークや機器の制御をコントローラで集中管理する
NFV（Network Function Virtualization）ネットワーク機器の機能を仮想マシンでソフトウェア的に実現。柔軟なネットワーク構成が可能
← S03:p.36-37

第3章

エンドユーザー向けの仮想化技術

第2章との違い

第2章の仮想化技術は基本的にインフラストラクチャ側（企業・大学等の組織が利用）。エンドユーザー向け仮想化技術は、エンドユーザーが利用するPCやスマートフォン上で仮想化技術を活用するもの。主にリモートワークへの活用が目的。← S03:p.43

エンドユーザー向け仮想化技術の共通メリット

メリット内容
フレキシビリティ自分のデバイスから時間・場所を問わずアクセス可能。異なるデバイスからでも仮想環境にログインして利用できる
セキュリティデータをセキュリティの高い場所に保存可能。ローカルへのデータ保存が不要でリスク低下
デバイス非依存仮想環境はクラウド上で実行されるため、エンドユーザーのデバイス性能に依存しない
バックアップと復元クラウド上で定期的なバックアップが実行され、データ喪失時はバックアップから復元できる
マネージャビリティシステム全体の管理は管理者（組織）が一括実施。セキュリティポリシーやアプリを管理者から配信できる
← S03:p.44-45

アプリケーション仮想化

定義

アプリケーションを仮想化してサーバに格納し、クライアントからのアクセス時に必要な部分だけを提供する技術。実行ごとに独立した仮想環境が起動されるため競合を防げる。← S03:p.46

音声補足

通常のアプリケーションはインストール時にシステム内に設定ファイルや実行ファイルを配置するため、同時に複数使用すると競合が発生することがある。アプリケーション仮想化では実行ごとに独立した仮想環境を起動するので、複数ユーザーが同じアプリを使っても競合しない。← S05

第4章

バックアップの重要性

定義

データのバックアップとは、計算機上に存在するデータやファイル、設定情報などを複製して保管すること。障害やトラブル等によりデータが紛失・破損した場合でも復旧・回復するために実施する。重要なデータの保全や安全性の確保が目的。← S03:p.59

バックアップはなぜ必要か

理由内容
データ損失の防止不測の事態によるデータ消失を防止。バックアップがあれば復旧できる
ビジネス継続性の確保データ消失による業務の停滞・損失を極小化し、システムの停止や破壊を防止
法的要件の順守法令によるデータ保管義務に対応
経済的損失の防止データ消失や破損が経済的損失につながることが多い。影響を最小限に抑える
← S03:p.61

バックアップの4種類

種類概要メリットデメリット
フルバックアップある時点のデータ全体をバックアップ完全性が高い / 復旧時間が短い / 操作が簡単毎回全データをコピーするためストレージ消費が大きい / バックアップ時間が長い
差分バックアップフルBK以降に変更されたデータ全てをバックアップ。
月曜フル→火曜は月-火の差分、水曜は月-水の差分		フルよりデータ量が少ない / 復旧が比較的簡単（フル+最新差分の2ファイルのみ）管理がフルより煩雑 / 日が経つにつれ差分ファイルが大きくなる
増分バックアップ前回バックアップ以降の変更のみをバックアップ。
月曜フル→火曜は月-火の差分、水曜は火-水の差分のみ		最もデータ量が少ない / 処理が高速（前回との差分のみ）管理が複雑 / 復旧にフルBK+全ての増分ファイルが必要 / 1つ欠けると復元不可
ミラーバックアップデータ変更があると別ストレージへ同時書き込み。
オリジナルから削除されるとミラー先からも削除される		復旧が最も容易 / 保管ストレージの物理的分散ができるストレージ負荷が大きい / 災害時にオリジナルとバックアップが同時に失われる / 同容量が必要
← S03:p.62-65
差分 vs 増分 の覚え方

差分バックアップ：フルBKからの累積差をその都度保存する。「ベース（フルBK）からの差」
増分バックアップ：前回との差のみ保存。「インクリメンタル（毎回少しずつ増える）」
→ 差分は復旧が簡単（2ファイル）、増分はデータ量が最少だが復旧が複雑（全ファイル必要）

条件推奨

監視と管理

重要原則

バックアップを取っても復旧できなければ無意味。バックアップを有効に利用するためには監視と管理が必要不可欠。← S03:p.70

管理項目内容
ログの監視バックアップジョブのログを監視し、成否やエラーを確認。実行されていない・エラーが発生した場合は対処が必要
テストリストア（復旧試験）バックアップを使って実際にデータを復元できるかテスト。定期的に実施しバックアップデータの復旧性を確認する
保管場所・期間の管理安全な場所への保存・アクセス権設定・保管期限の管理。保管期限が定まっている場合は必要以上の期間保管しない
自動化手動実行による人為的ミスを抑制。定期的に最新の状態を保つ
← S03:p.70-71

バックアップの検証

検証方法内容
ファイルチェックサムの比較オリジナルファイルとバックアップファイルのチェックサムを比較。不一致はバックアップに問題があることを示す
ファイル復元テスト実際にファイルを復元し、オリジナルと比較してバックアップの正常性を確認
システムの復元バックアップからシステムを復元し、正常に動作するか確認
定期的なテストリストア復旧までの時間・手順を確認し、問題があれば修正。「テストリストアまで実施して初めてバックアップが有効」
← S03:p.72-73

バックアップデータの保管場所

ポイント内容
オフサイト性システムと切り離された別の場所（オフサイト）に保管。災害・人災による被害を防ぐ。例: アジア・ヨーロッパ・オーストラリアの3か所で保管
安全性盗難・喪失リスクが少ない場所。データの暗号化・アクセス制御などのセキュリティへの配慮も必要
アクセス性復旧が必要な時に迅速にアクセスできる場所。バックアップデータの取り出しや復元作業が行いやすい場所
コストオフサイト保管や高度なセキュリティ対策を施す場合の運用コストも考慮。一般的に厳重な対策ほどコストが高い
法規制・規約国・地域・業界によっては保管場所に法的規制や規約が存在する。これらに則った選定が必要
← S03:p.76-77

暗号化とアクセス制限

バックアップデータの暗号化

共通鍵暗号方式公開鍵暗号方式
暗号化・復号に使う鍵1つの鍵で暗号化と復号を行う暗号化と復号に別の鍵を使う

鍵の管理難しい比較的簡単

どちらでも鍵をなくした場合はデータにアクセスできなくなるため鍵の管理は最重要。バックアップソフトウェアの暗号化機能やディスク・テープの暗号化装置を使う方法もある。← S03:p.78

アクセス制限

  • アクセス権限をもつアカウントのみに制限し、権限は必要最小限に設定
  • 物理的なセキュリティも考慮（施設・設備のセキュリティ・防火対策・防水対策）
  • アクセスログを定期的に監視し不正アクセスや不正利用を早期発見
  • バックアップストレージを本番環境から分離（オフサイト）する
  • バックアップソフトウェアやバックアップサーバの動作ログを監視し不正操作・故障を早期発見
← S03:p.79-81

バックアップの自動化

手動でバックアップを行うとヒューマンエラーの可能性がある。正しく設定された自動化によりミスの可能性を抑えることができる。事前に目的・データ保護方針・スケジュール・バックアップ先・セキュリティ等を明確にすることが重要。← S03:p.82

自動化のメリット
手動作業が不要なので作業コストを抑えられる
定期的で正確なバックアップを取得できる
時間指定が可能なため業務時間を避けて実施可能
自動実行のプロセスや結果についてログを取得できる
バックアップの実行状況を一元的に管理できるためトラブルシューティングが容易

📚 補足B: 知っておくべき背景知識

GDPRの罰則体系（2段階制） [外部補足・確認済: EU公式 Art.83]

GDPRはGeneral Data Protection Regulationの略で、2018年5月25日に施行。第83条で罰則が2段階に規定されている:

  • Tier 1（軽微な違反）: 最大1,000万ユーロ または 全世界年間売上高の2%のいずれか高い方。適用例: DPO（データ保護責任者）の未任命、処理記録の不備、データ侵害の未通知等
  • Tier 2（重大な違反）: 最大2,000万ユーロ または 全世界年間売上高の4%のいずれか高い方。適用例: 基本原則の違反、同意なきデータ処理、データ主体の権利侵害等

実際の高額事例: MetaへのTier 2罰金12億ユーロ（2023年、アイルランドDPC）、AmazonへのTier 2罰金7億4,600万ユーロ（2021年）等。

VMware vSphere: 2023年にBroadcomがVMwareを約610億ドルで買収。製品ラインの整理と価格体系の変更が進んでいる（2024年現在）。
Microsoft Hyper-V: Windows Server 2008以降の一部エディションに標準搭載（追加コストなし）。中小規模のシステムで広く使われる。

デスクトップ仮想化（VDI）製品の現状 [外部補足・確認済: Citrix公式・TechTarget]

XenDesktop → Citrix Virtual Desktops: 2007年にCitrixがXenSourceを買収し2008年にXenDesktop v2.0をリリース。2018年にCitrix Virtual Apps and Desktops（CVAD）に改名。
VMware Horizon: 旧VMware View。2023年のBroadcom買収後はOmnissa Horizonとして分社化された。

Dell EMCとNetAppの現状 [外部補足・確認済: Wikipedia / Dell Technologies公式]

スライドに記載の「EMC」は現在Dell Technologiesのストレージ部門として存続。Dell Inc.が2016年9月に670億ドルで買収（当時IT業界史上最大の買収）し「Dell EMC」に、2019年頃から「Dell Technologies」ブランドに移行。
NetAppは独立系ストレージ専業大手として現在も存続。クラウドネイティブなデータ管理を推進中。

3-2-1バックアップルール [外部補足]

バックアップのベストプラクティスとして知られる「3-2-1ルール」: 3つ以上のコピーを保持、2種類以上の異なる媒体に保存、1つはオフサイト（物理的に離れた場所）に保管する。本章で学んだオフサイト保管・地理的分散の実践的な指針。

🎯 補足C: 試験・レポートで重要な観点
  • 冗長化の4種類（ハードウェア・ソフトウェア・配置・データ）と5種の構成方式の特徴比較（ホットスタンバイvsコールドスタンバイ、アクティブ・アクティブ、マスター・スレーブ、マルチマスター）は必出。それぞれのコスト・復旧速度のトレードオフを説明できること
  • サーバ仮想化3種類（ハイパーバイザ型・ホストOS型・コンテナ型）の仕組みの違いと各型の特徴を図示して説明できること
  • 差分バックアップと増分バックアップの違いは特に頻出。「差分はフルBK以降の変更全て」「増分は前回BK以降の変更のみ」という違いと、復旧時に必要なファイル数の違い（差分=2ファイル、増分=全ファイル）を理解する
  • バックアップはテストリストアまで実施して初めて有効：「取るだけでは不十分」という点は試験頻出
  • システム停止による4種類の被害（経済的損失・社会的影響・信用の失墜・法的ペナルティ）を具体例とともに説明できること
  • GDPRとBCP：組織・法律的な観点から問われることが多い。GDPRはEU域外企業にも適用される点が重要
  • 共通鍵暗号vs公開鍵暗号の速度・鍵管理の比較も出題されやすい
🔗 補足D: 関連概念・次回への接続

RPO / RTO [外部補足]

バックアップ計画に関連する重要な概念。RPO（Recovery Point Objective：目標復旧時点）は「どの時点のデータまで失うことを許容するか」の目標値。RTO（Recovery Time Objective：目標復旧時間）は「障害発生から何時間以内に復旧するか」の目標値。BCPの策定においてこれらを設定することが必要。バックアップの頻度は主にRPOによって決定される。

クラウドと仮想化の関係 [外部補足]

AWS・Azure・Google Cloudのようなクラウドサービスは、本章で学んだ仮想化技術（サーバ仮想化・ストレージ仮想化・ネットワーク仮想化）を組み合わせてサービスとして提供している。IaaS（Infrastructure as a Service）はこれらを抽象化して利用者に提供するもの。エンドユーザー向けのVDIもクラウド上で提供されるDaaS（Desktop as a Service）として提供されるケースが増えている。

Docker/Kubernetesとコンテナ型仮想化 [外部補足]

Dockerはコンテナ型仮想化の代表的実装。アプリケーションとその依存関係をまとめた「Dockerイメージ」を作成し、どの環境でも同じように動かせる（"Build once, run anywhere"の思想）。Kubernetes（K8s）はコンテナのオーケストレーションツールとして広く使われ、第4章の「バックアップオーケストレーション」と概念が共通している。

ランサムウェアとバックアップ戦略 [外部補足]

近年増加しているランサムウェア攻撃では、ネットワーク接続されたバックアップも暗号化の対象となる。これに対する「オフライン（エアギャップ）バックアップ」の重要性が増している。本章で学んだオフサイト保管・ストレージの本番環境からの分離はランサムウェア対策としても機能する。

SDN/NFVとネットワークの未来 [外部補足]

SDNやNFVの概念は、インフラ全体をコードで管理する「IaC（Infrastructure as Code）」やクラウドネイティブなアーキテクチャの基盤となっている。Terraform・AnsibleなどのIaCツールとSDNを組み合わせることで、大規模なネットワーク環境を効率的に管理することが可能。
