第1章

問題意識を高めよう

問題意識の重要性

章の学習目標

プロジェクトを始めるにあたりプロジェクトマネージャーとしてどのような意識になっておくべきかについて理解し、説明できる ← S03:p.11

あなた（PM）のプロジェクトに対する目的がブレたら失敗する。 ← S03:p.12

ぶれないための3つの意識

意識内容失敗例
問題意識を高める 「他人事」を「自分事」に。自分がやらなければいけない・達成したいと思う やれと言われたのでやってるだけ → モチベーション低下 → 欠点に気づかない
タイミングを見極める 「機が熟したか」を考えてから着手する 時期尚早・関係者揃わず・資金難の状態で着手 → 途中でストップ
意義を見つめる 会社にとって・地域にとって・地球にとってやる価値があるかを確認する 意義なき着手 → プロジェクトへの前向きな取り組みが生まれない

← S03:p.12

🔊 音声補足

問題意識が高くなかったら失敗が目に見えている例：「家を建てる気はなかったが金利が低くなったのでなんとなく」「カレーを作る気はなかったが玉ねぎとにんじんがあったのでなんとなく」「システムを作る気はなかったが作れと言われたので」——想像するだけでも、あまりおいしいカレーが作れるとは思えない。狙っていかないと料理もうまくならない、同様にシステムも。 ← S05-A, S05-B

大規模プロジェクトへの参画は他の要因にも左右するほどの大きな取り組みになる。そのため以下の3点が求められる。 ← S03:p.14

  • 専任化——ある一定期間、プロジェクトに専従する（兼務しない）
  • 専門化——その道のスペシャリストとして携わる
  • 評価基準の設定——「成功するとこうだ・失敗するとこうだ」という評価基準を作った上で参画する。そうしないと成功しているのか失敗しているのかわからなくなる

過去を振り返る

格言

「過去の失敗は最高の先生」 ← S03:p.17

  • 過去に同様の取り組みをして失敗していないか（例: 3年に1回のダイエットプロジェクト）→ 失敗の要因は何だったか？ → 今回それを克服するにはどうするか？
  • これまで避けられてきたプロジェクトではないか → 取り組みを避ける要因（例: ブランドイメージに反する・反対勢力がいる）が存在した場合、その理由を取り除くことができる

← S03:p.17

🔊 音声補足

プロジェクトでも何らかの要因があって取り組みを避けることがある。例えば「こういうことはうちの会社のブランドイメージに反する」「こういうことをやろうとすると必ずどこかから反対がある」。そういう場合でも、同様に失敗するのではなくその理由を取り除くことができるので、過去の失敗事例を振り返っておくことが重要。 ← S05-A, S05-B

他を参考にする

前提

「あなたは特殊ではない」——探してみると世の中には同じような事例がある。自分たちが考えつく課題は社会の一部で発生するもので、誰かが同じ経験をしていることが多い。 ← S03:p.18

  • セミナー・同業の交流会に参加して同様の取り組み事例を学ぶ
  • 新聞・雑誌・本に目を通す（料理・スポーツ分野でも有効）
  • TTPS（徹底的にパクって進化）——他のやり方を真似する「P（パクる）」だけでなく、さらに良くして自分たちのやり方として適用する「S（進化）」が重要 ← S03:p.18
  • 先端でなく先進——世の中の1番でなくてよい。先取りしすぎると不安定要因が多くなり失敗する。どこかがやっている経験を元に取り組むことが成功確率を高める ← S03:p.18
  • 他の現場をその目で見に行く——先輩に学ぶことは恥ずかしいことではない。同業・異業種の他社を訪問し、課題・成果・うまくいった点/いかなかった点を聞く ← S03:p.19
  • プロジェクトの途中でも意見交換できる関係性を保つ ← S03:p.19
🔊 音声補足

TTPSのSが重要。どこかがやったことをさらに良くして自分たちのやり方として適用していく。また「先端」ではなく「先進」——例えば技術的な要因でも不安定要素が高い最先端技術はリスクが大きい。1番でなくてもよいものであれば先進として、どこかがやっている経験を元にすることで成功確率が高まる。 ← S05-A, S05-B

章まとめ
  • プロジェクトを始めるにあたり、リーダーとなるあなたは十分に熱くなっただろうか？
  • あなたが熱くならなければ、これからのプロジェクトの立ち上げは成功しない ← S03:p.20
第2章

規模の見極め

章の学習目標

プロジェクトに取り組む際の規模の見極めについて理解し、説明できる ← S03:p.23

プロジェクトには与えられた条件のもと、適正なサイズ・レベルがある。条件が変われば変わるが、条件が同じなら適正なレベルが存在する。 ← S03:p.24

松・竹・梅プランで目標を設定する

プラン意味詳細
🌲 松 上位目標 できたらいいな、という高いレベルの目標
🎋 竹 必達目標 必ず達成すること。ここができないとプロジェクトとして困るレベル。詳細まで徹底的に詰める
🌸 梅 最低限 動かないと大変なことになる最低ライン。本当は竹を達成すべきだが少なくとも梅は必要

← S03:p.24

必達目標（竹）については、以下を詰めること。

  • 要件として盛り込むべきことの漏れはないか
  • リスクは何か（期間・コスト・技術 etc.）

← S03:p.24

🔊 音声補足

竹のレベルを達成できなかったとしても、せめて梅のレベルは動かないと大変なことになる——というレベル付けをする。この授業では「松竹梅」という日本的な表現を使っているが、このようなレベル別のプランを作ることが重要。 ← S05-A, S05-B

スコープ・期間・資金の見極め

スコープ

定義

スコープ（scope）: プロジェクトの狙いとする領域。スコープを決めなくては目的地がないのでスタートできない。 ← S03:p.25

🔊 音声補足 — スコープの例

チキンカレーでも「一人前500円のチキンカレー」なのか「800円のチキンカレー」なのか「一人前1万円のチキンカレー」なのかでレベル感が異なる。システムも同様に、どれくらいの規模・ステップ数・投資金額のプロジェクトかを最初に決めておく必要がある。 ← S05-A, S05-B

期間

なぜ期間が重要か

① コストにつながる——期間が延びると大体の場合コストが積み上がっていく
② 効果につながる——長くかければ効果が出やすい傾向もあるが、かけすぎると失敗するリスクもある
③ 効き目の高いタイミング——早すぎても遅すぎてもダメ。成果が出るタイミングを見極めて実行する ← S03:p.26

例コスト観点効果観点タイミング
カレーを作る 3日もかけられない（人件費） 夕飯に食べたい（いつ提供するか） 早すぎ=鍋が使えない
遅すぎ=夕飯を逃す
家づくり 2年もかけられない（人件費等） 受験勉強に子供部屋が必要 早すぎ=頭金が用意できない
遅すぎ=狭くて不満が爆発

← S03:p.27-28

資金（投資対効果分析）

定義

投資対効果分析（ROI分析）: 投資額と効果額を比較する分析。重要なポイントは効果を定量的（お金に換算）に算定すること。お金対お金で比較しないと投資に対して効果が見合っているかどうかがわからない。 ← S03:p.29

  • 投資額はプロジェクト計画の中で積み上げて算定できる
  • 効果額を金額に換算するのは難しいが必ず行う
  • さらに現実的な分析: 現在価値に直した評価（リスク率を換算して割り引く） ← S03:p.29
外部補足 — ROI・NPVの計算式（Web確認済み）

ROI（投資利益率 / Return on Investment）
ROI(%) = (効果額 − 投資額) ÷ 投資額 × 100
例: 投資100万円 → 効果150万円のとき、ROI = 50%。プロジェクトの採算性を示すシンプルな指標。

NPV（正味現在価値 / Net Present Value）: 投資による将来キャッシュフローを、割引率（リスク率）を使って現在価値に換算した総和から投資額を差し引いた値。
NPV = Σ [ 各年のキャッシュフロー ÷ (1 + 割引率)^年数 ] − 投資額
判定基準: NPV > 0 なら投資価値あり / NPV < 0 なら投資価値なし
割引率（リスク率）は一般的に資本コスト（WACC）を使用。実務では4〜7%が目安とされる。
例: 1年後100万円の効果・割引率5%の場合 → 現在価値 = 100 ÷ 1.05 ≈ 95.2万円

ROI vs NPV の使い分け: ROIは計算が簡単で直感的。NPVは「将来の不確実性・時間価値」を考慮した精緻な分析。スライドで講師が「現在価値に直した評価」と言っているのはNPVによる分析を指す。

例資金の見極め方
カレーカレーに求める価値にふさわしく、しかも通常の夕食の予算の2倍以内
家づくり頭金と月々の支払いが可能な範囲か検証（無理な返済計画はプロジェクト失敗につながる）
システム得られる効果より少ない投資であり、かつ経営への影響を最小限にする額か検討

← S03:p.30

人材・リスク・予想される喜び

原則

最も重要な資源、それは人である。 ← S03:p.31

人を集める際に検討すべき3点：

  • どんな役割が必要か——システムの要件から役割を書いてみる。システムの規模から組織と人数を書いてみる
  • どこにその人はいるか——社内を含め調達可能なところを探しておく。社内か社外プロフェッショナルか
  • どうすれば採用できるか——ベストメンバーをそろえるための作戦を立てる ← S03:p.31

予想される壁（リスク）を顕在化する

プロジェクトをスタートする前に、越えなければいけない大きな問題を書き出して共有し見えるようにする。 ← S03:p.32

リスクの種類確認すべき内容
技術リスク 新しい技術・新しい通信手順が必要か。どれくらい難しくてどうやったら克服できるか
費用リスク 費用がプロジェクト途中で増えるとしたら、その要因は何か
期間リスク 期間が延びるとしたら要因は何か。どれくらいかかりそうか
🔊 音声補足 — 顕在化の方法

顕在化の方法として最も基本的なのは「書いておくこと」「書いて共有すること」。潜在的なリスクを書き出して関係者間で共有することで、実際に実行した時の対処の仕方が変わってくる。 ← S05-A, S05-B

予想される喜びの大きさを考える

重要な視点

人はメンタルな動物。プロジェクトが成功した時にどんな喜びがあるかを想像し、関係者と分かち合う喜びのシーンを思い浮かべて描いておく。 ← S03:p.33

関係者（上司・仲間・お客様・知り合い・家族）のプロジェクトが終わった後の状態を描いておくと、より積極的にプロジェクトに携わろうという気持ちになる。 ← S03:p.33

ラフなプロジェクト計画書を一人で作る

定義

エレベータートーク: エレベーターの中にいる間（約1分以内）でプロジェクトの要点を説明できること。自分の頭の整理 = プロジェクトの要点が言語化された状態。 ← S03:p.34

  • ねらい（何のために）
  • 取り組みテーマ（何を・どのように）
  • 必要な資源（人・モノ・金）

← S03:p.34

章まとめ
  • プロジェクトを始める前に、自分でラフな（粗い）プロジェクト計画を作ってみよう
  • プロジェクトの全体像を知ることで自分の取り組む内容が明確になってくる ← S03:p.35
第3章

関係者を組み立てる

章の学習目標

プロジェクト推進に必要な関係者の組み立てについて理解し、説明できる ← S03:p.38

ステークホルダー（利害関係者）の図式を作る

定義

ステークホルダー（利害関係者）: プロジェクトに関係する全ての人・組織。 ← S03:p.39

役割例（カレー）例（家づくり）
スポンサー（オーナー）お父さんあなた
プロジェクトマネージャーあなた施工主
メンバー長女・長男大工・電気屋・内装屋・造園業者
利用者家族全員家族全員

← S03:p.40-41

役割・責任・権限の関係

概念定義カレーの例
役割 何々をする役割 玉ねぎを炒める役割
責任 役割を果たす責任 玉ねぎを期限までにアメ色まで炒める責任
権限 何々を決める権限 コンロ・鍋・材料を使う権限。炒め具合（止める判断）の権限

← S03:p.42

原則

与えられた「役割」を「責任」をもって行う。それに必要なのが「権限」。 ← S03:p.43

  • 与えられっぱなしの役割はない——それぞれが責任をもってやる
  • 成果につなげるために、権限を積極的に使う。権限を使わないと役割を果たせなくなる ← S03:p.43

会議体とその役割

会議体役割例
家族会議・経営会議 投資の承認 プロジェクト開始の意思決定
ステアリング・コミッティ プロジェクトの方向性の決定 大きな仕様変更・スタート時期変更・投資金額変更
定例進捗報告会 プロジェクト進捗の共有 日次・週次・月次。リリース前日は1時間ごとにリーダー会議

← S03:p.44

用語解説

ステアリング・コミッティ（Steering Committee）: ステアリング（舵）＋コミッティ（委員会）。プロジェクトの方向性を決定する会議体。例: 大きな部分での仕様変更・スタート時期変更・投資金額の変更などを決定する。 ← S03:p.45

🔊 音声補足 — 定例会の刻み方

定例会は場面場面によって刻み方が変わる。例えば「関係者のリーダーで月に1回・毎第一月曜日」「関係者のサブリーダーレベルで毎週金曜日午後5時（5時から会議）」「チーム内で9時から10分の朝会」のように、テンポを決めておくと共有のレベルが難なくできる。特殊な場合——例えばシステムのリリース前日は1時間ごとにリーダー会議が行われることもある。 ← S05-A, S05-B

メンバー表を作る

時間軸に沿って、どの時期にどの役割のメンバーが何工数必要かを示した表。コスト・組織・作業スペースの見通しにも使う。 ← S03:p.47

🔊 音声補足 — メンバー表の例

カレーの例: 「炒める・煮る」担当は最初（炒め）と最後（煮る）に工数が集中。「材料を切る」は最初だけ（後半はゼロかも）。「味を整える」は途中と後半に出てくる。
家づくりの例: 大工さんは最初から最後まで必要。基礎工事は最初のフェーズだけ集中（後は不要）。内装業者は最後のみ活躍。このように役割ごとに工数の山が異なる。 ← S05-A, S05-B

章まとめ
  • プロジェクトにおいてどんな役割と責任があるのかを書き出す
  • 役割のところに加えたいメンバーの名前（または必要なレベル）を書く
  • どんな会議を作るとうまく運営できるかを考え、あらかじめ計画を立てておく ← S03:p.50
第4章

プロジェクトの立ち上げ

章の学習目標

プロジェクトの立ち上げプロセスについて理解し、説明できる ← S03:p.53

立ち上げまでのプロセス（全体像）

中期計画・年度計画の確認

定義: 中期計画

中期計画: 3〜5年の期間で、組織（家・会社・部門等）が取り組む課題と方向性を書いたもの。1年ではできないことや短期ではできない大きな方向性を示す。 ← S03:p.55

中期計画のポイント:

  • 組織としての課題認識——何が問題だと思っているか
  • 組織としての取り組み——どう解決しようと思っているか
  • 組織としてのスケジュール——どの順番で、いつ取り掛かろうと思っているか ← S03:p.56
🔊 音声補足 — 中期計画の例

中期計画の取り組む方向性の例: 「野菜中心の食生活にもっていく」「両親と一緒に暮らす」「品質の優れた定番商品ブランドを作る」など。これを中期的に3〜5年の間に成し遂げようとする方向性などを決めることが中期計画。 ← S05-A, S05-B

定義: 年度計画

年度計画: 今年度はどんなことに取り組むかのテーマを、人・お金の割り振りを具体的に考慮して書いたもの。中期計画の年計画へのブレークダウン。 ← S03:p.57

🔊 音声補足 — 年度計画の展開例

「両親と同居する」という中期計画を例にすると:
・昨年: 両親と家族の意向を固める（家族旅行）
・今年: 土地を決める（今の土地の処分）
・来年: 設計と発注
・再来年: 家の完成と引越し
↑それぞれもプロジェクト。中期計画の達成のために各年度に目標を設け、PDCAで回していく。 ← S05-A, S05-B (S03:p.59)

年度計画のポイント（中期計画をより具体的に）:

  • 組織としての課題認識をより詳細に確認する
  • どの作戦・施策でどう解決するか
  • どの順番で、いつ取り掛かるか（資源の割り振り）← S03:p.58

参画部門への根回し・決議機関への上申・キックオフ

参画部門への根回し

現場に行って「本当に必要だ！」という想いを取り付ける。 ← S03:p.60

  • 確認のため——本当にこの課題は現場で起きている問題なのか確認する
  • 味方を増やすため——一緒に同じ問題を見て「あの辺課題ですよね」と認識を共有する → プロジェクトの仲間が増える
  • メンバーに参加してもらうため——「こういう解決策はどうですか？」と提示することでメンバー作りの根回しができる ← S03:p.60
🔊 音声補足

リーダーだけが「やろう！」と声を上げても、プロジェクトのメンバーや参画すべき部門の方々が実際に協力してくれないとプロジェクトは進まない。現場に行くことで課題の共有ができ、「自分もこの辺が課題だと思っていた」「我が社にとってこの課題の解決は非常に重要だ」という気運が生まれる。 ← S05-A, S05-B

決議機関への上申・提案

定義

決議機関: 経営会議などの意思決定機関。ここでの決定がないと、正式なプロジェクトとしての活動ができない（組織・人・お金・権限が付かない）。 ← S03:p.61

中期計画・年度計画があり、関係部門への問題意識共有ができた段階で、決議機関に提案する。例: 家族会議・町内会議・職員会など。 ← S03:p.61

🔊 音声補足

課題に思っていて「こういうのはやったらいいよね」と言っているだけではサークル活動。企業や組織で実際に実行するには、正式に組織として意思決定し、資源（人・お金・情報・権限）を投入する必要がある。可決されれば実行に移り、否決されればそのプロジェクトはスタートしない。 ← S05-A, S05-B

キックオフ

定義

キックオフ（Kickoff）: プロジェクトの最初の会合。全メンバーが出席する。 ← S03:p.62

構成内容重要ポイント
① オーナーから趣旨説明 意図・期間・ゴール・投資額・効果見込みの説明 トップがどういうことを目指しているかをメンバー全員が把握する
② プロジェクト概要説明 PMが目的・進め方・期間・ステップを説明 PMの最初の重要な仕事
③ メンバー自己紹介・意気込み 関わるメンバーが自己紹介と意気込みを話す 初めて会うメンバー間のコミュニケーション促進
④ 懇親会（できれば） 飲み食いだけでなく意見交換・心のバリアを取り払う 初対面の人は疑心暗鬼からスタート → 懇親会で最初のバリアを取る

← S03:p.62

章まとめ（第4回全体まとめ）
  • プロジェクトの中期的視点からの位置づけを確認する——中期計画の中でのやるべき理由を確認
  • 周りの理解を得ながら、協力の得られる体制を作る——関係部署への根回し・現場訪問
  • 議決を得て正式に組織として決定し、キックオフしてプロジェクトスタート ← S03:p.63

📚 補足B: 知っておくべき背景知識
  • [外部補足] PMBOK（Project Management Body of Knowledge）とは: PMI（Project Management Institute）が策定したプロジェクトマネジメントの知識体系ガイド。1996年初版から始まり、現在は2021年発行の第7版まで改訂されている。
    ・第6版まで: プロジェクトの進行を「立ち上げ」「計画」「実行」「監視・コントロール」「終結」の5つのプロセス群に分類。立ち上げプロセス群は主に「①プロジェクト憲章の作成」「②ステークホルダーの特定」の2プロセスで構成。
    ・第7版: 5プロセス群の代わりに「12の原則」と「8つのパフォーマンス領域」を導入。「何をするか」よりも「なぜするか」という原則ベースの思想に転換。
    ・本講義の「立ち上げプロセス」はPMBOK第6版の「立ち上げプロセス群」に相当する。
  • [外部補足] プロジェクト憲章（Project Charter）（PMBOK第6版）: 「プロジェクトを正式に認可し、プロジェクトマネージャーに組織資源を活用する権限を与える文書」。記載内容: プロジェクトの目的・背景・スコープ・予算・主要マイルストーン・ステークホルダー・PMの権限など。この文書が承認された瞬間からPMは公式にプロジェクトを動かす権限を得る。本講義の「決議機関への提案書＋可決」に相当する。
  • [外部補足] ステークホルダー分析の手法（PMBOK）: ステークホルダーを特定した後、「権限（影響力）の強さ × 関心度の高さ」の2軸マトリクス（Power/Interest Grid）でグループ分けし、対応戦略を決める。グループごとに「重点管理」「情報提供」「満足維持」「観察」の優先度を設定する。ステークホルダー登録簿（識別情報・評価情報・分類情報）を作成して文書化する。
  • [外部補足] KJ法の詳細: 文化人類学者・川喜田二郎（Jiro Kawakita）が1960年代に考案したことから頭文字をとって「KJ法」と呼ぶ。①情報をカードに書き出す → ②類似カードをグループ化 → ③各グループに見出しをつける → ④グループ間の関係を図示する、というステップで問題の本質を発見する。プロジェクト管理の問題分析・要件定義・アイデア整理に広く使われる。
🎯 補足C: 試験・レポートで重要な観点
  • プロジェクトマネージャーの問題意識（自分事化）がなぜ重要かを説明できるか → 「他人事にするとモチベーション低下・欠点に気づかない・失敗に直結」
  • 松・竹・梅プランの定義と、なぜ「竹（必達目標）」を詳細まで詰めるのかを説明できるか
  • 投資対効果分析で効果を定量化（金額換算）することの意義を説明できるか。ROI（投資利益率）とNPV（現在価値ベース）の違いも押さえておくと尚良い
  • スコープ（scope）の定義: プロジェクトの「狙いとする領域」。スコープなしにはスタートできない
  • ステークホルダーの4種類（スポンサー/オーナー・PM・メンバー・利用者）と役割の違いを説明できるか
  • ステアリング・コミッティとは何か——「舵（ステアリング）取り委員会（コミッティ）」＝プロジェクトの方向性を決定する会議体。決定事項の例（仕様変更・スタート時期変更・投資金額変更）も覚えておく
  • 役割・責任・権限の三角形——「役割を果たすために責任があり、それに権限が必要」。権限を積極的に使うことが役割遂行の条件
  • 立ち上げプロセスの流れ（4ステップ）: 中期計画確認 → 年度計画確認 → 参画部門根回し → 決議機関上申 → キックオフ
  • キックオフの4構成: ①オーナー趣旨説明 ② PM概要説明（PMの最初の重要な仕事）③メンバー自己紹介・意気込み ④懇親会
  • 「過去の失敗は最高の先生」「先端ではなく先進」「TTPS（徹底的にパクって進化）」「人はメンタルな動物」「エレベータートーク」——講師独特のフレーズの意味と文脈を押さえる
  • 定常業務 vs プロジェクトの違い: プロジェクトは期間が決まっており特殊。日々継続的に発生する問題は定常業務・専門部署での対処が適している場合もある
