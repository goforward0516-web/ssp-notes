CH 01

サーバシステムの基礎

学習目標
  • クライアント・サーバシステムの基本的な概念について理解し説明できる
  • サーバシステムとネットワークの関係について理解し説明できる

1-1 サーバとクライアント

サーバ（Server） とは、ネットワーク上で他のコンピュータに サービスを提供するコンピュータのこと。← S03:p.6

サーバにはサービスを受ける相手先 — クライアント — が存在する。 スマートフォンやパソコンでWebサイトを閲覧したり、メールを受け取る端末は すべてクライアント側にあたる。

🔊 音声補足（池田先生）
サイバー大学のCloud Campusを閲覧している画面もWebサーバからサービスを受けている。 日常的に使っているスマートフォンはクライアントの典型例。会社や学校内にも、 プリンタサーバ（印刷を管理するサーバ）やファイルサーバ（社内ファイル共有）など、 多数のサーバが稼働している。← S05-A/B

サーバの役割はサービスの種別で決まる

カテゴリ代表的なサーバ説明
インターネット公開WebサーバWebサイトの情報をクライアントに提供
インターネット公開メールサーバメールの送受信を管理
組織内使用プリンタサーバ社内・学校内の印刷管理
組織内使用ファイルサーバ組織内でのファイル共有
インターネット接続DNSサーバドメイン名とIPアドレスの対応付け

← S03:p.7, p.10

ポイント: ネットワーク上でサービスを提供するために多くのサーバが存在する。 1つの組織内に複数のサーバが稼働しているのが一般的。

1-3 クライアント・サーバシステム

クライアント・サーバシステム とは、クライアントとサーバで構成された、 現在の一般的なコンピュータシステムの方式。← S03:p.8,9

最重要ポイント: サーバはクライアントからの リクエスト（要求）を受けてから処理を開始し、 サービスを提供する。サーバが一方的・自発的に動作するわけではない。
[クライアント] ──(1) 要求（Request）──→ [サーバ] [クライアント] ←──(2) 返答（Response）── [サーバ]
🔊 音声補足（池田先生）
Cloud Campusへのアクセスを例にすると: ブラウザ（クライアント）が Cloud Campusの情報を要求 → サーバが処理 → 返答としてページが表示される、 という流れ。このやり取りのプロトコルとしてHTTP・HTTPSが使われる （第2回第3章参照）。← S05-A/B

1-4 ファイアウォールとDMZ

ファイアウォール（Firewall）

ファイアウォール（「防火壁」の意）は、 必要な範囲だけインターネットとの通信を許可する機器・ソフトウェア。 ← S03:p.12

🔊 音声補足（池田先生）
社内ネットワークをインターネットにそのまま接続することは基本的にしない。 ファイアウォールを通して、プロトコル（通信の規約）に基づいて 許可・遮断の設定をする。ハードウェアの場合もあれば ソフトウェアで実現する場合もある。← S05-A/B

DMZ（De-Militarized Zone）「非武装地帯」

DMZ は、インターネット上に「公開するサーバ」と 「公開しないサーバ」を分離したネットワーク構成。 ← S03:p.13

🔊 音声補足（池田先生）
DMZを設けることで、万が一公開サーバへの攻撃があってもDMZ内の影響にとどまり、 社内ネットワークは別のネットワークに分離されているため守れる構成になっている。 公開サーバの代表例はWebサーバ・DNSサーバ・メールサーバ。 ← S05-A/B

1-5 サーバ構築の検討事項

設置場所の選択

形態内容
オンプレミスサーバ本体・ネットワーク機器を自社で保有・運用
レンタルサーバ（共用）複数企業でサーバを共有
レンタルサーバ（専用）1社がサーバを丸ごと借用
VPSVirtual Private Server。仮想的に専用サーバ相当を構築（仮想化技術は第8回）
クラウドAWS（Amazon Web Service）・GCP（Google Cloud）・Azure（Microsoft Azure）等

← S03:p.15

🔊 音声補足（池田先生）
特に重要なのは初期設計。初期に導入した構成によってランニングコストや 更新費用が大きく変わる。機械は古くなり、ソフトウェアも更新が必要なため、 最初の設計が後々まで影響する。← S05-A/B

第1章 まとめ

  • サーバ＝「提供する側」。クライアントの要求を受けて動く（自発的ではない）
  • クライアント・サーバシステムが一般的なコンピュータシステムの基本方式
  • ファイアウォールで必要な通信のみを許可し、DMZで公開・非公開サーバを分離
  • 設置場所・仕様の選択はランニングコスト等に大きく影響する
  • LAN・WAN・DNS・IPアドレス・プロトコルなどネットワーク知識が重要
CH 02

Webサーバの仕組み

学習目標
  • Webサーバの仕組みについて理解し説明できる
  • Webサーバの構成について理解し説明できる

2-1 WebサーバはサーバシステムのCSS代表例

日常的に利用するサービスのほとんどでWebサーバが使われている。← S03:p.22

  • メール・検索サービス・SNS
  • ネットショッピング・オンラインバンキング

Web上でサービスを展開するメリット

メリット内容
いつでも・どこでもインターネット環境があれば利用できる
クライアント端末の制約が少ないブラウザベースであればOSの差異が少ない（Chrome: Windows/Mac/Linux対応）
高速化・大規模化が容易サーバ側のCPU・メモリ増強で対応でき、クライアント性能に依存しない
🔊 音声補足（池田先生）
Google ChromeはWindowsでもMacでも使える。クライアント側OSに依存しないのがWeb展開の利点。 高速化もサーバ側を増強すれば対応できるため、 クライアント側スペックに依存しない点が重要。← S05-A/B

2-2 Webサーバに求められる性能

3秒ルール: 3秒以内にページを表示できなければユーザーを取り逃す、という経験則。 ← S03:p.23
  • 表示が遅いとクライアント側から再読み込みリクエストが来て、サーバ負荷がさらに増す悪循環が生じる
  • 内容が即座に得られないと判断されてもユーザーは離脱する
🔊 音声補足（池田先生）
今のインターネットが高速化している時代では3秒は非常に長く感じる。 表示が遅いと再読み込みを連打されてサーバ負荷が急増するという別の問題も生じる。 WebサービスはクライアントのアクセスによってサーバのLoad（負荷）が流動的なため、 突発的な大量アクセスへの考慮も必要。← S05-A/B

2-3 Webアプリケーションと3層アーキテクチャ

現在の多くのWebサイトはWebアプリケーションでもある。 ショッピングサイトで住所を入力して商品を発注するように、 双方向のやり取りが発生し、裏側では様々なプログラムが稼働している。 ← S03:p.24

3層アーキテクチャ（3-Tier Architecture）

処理の種類ごとにサーバを分割し、負荷分散・システム変更の容易化を実現する設計。

🔊 音声補足（池田先生）
WebサーバとAPサーバを一緒にするケースもある。AP＝アプリケーションサーバの略、 DB＝データベースサーバの略。私たちの目には直接見えないが、 裏側でこれらが連動している。← S05-A/B

2-4 Webサーバ関連ソフトウェア

役割ソフトウェア名備考
WebサーバApache, Nginxオープンソース（無償）
WebサーバIISWindows Server付属（非OSSの商用製品）
APサーバTomcatオープンソース（無償）
DBサーバMySQL, PostgreSQLオープンソース（無償）
DBサーバOracle商用製品

← S03:p.26

多くのソフトウェアはオープンソースとして無償で利用できる。

2-5 LAMP構成

LAMP構成とはオープンソースのソフトウェア名を組み合わせた略称で、 Webサイト・Webアプリケーション構築によく使われる。 ← S03:p.27

すべてオープンソース → すべて無料で利用可能
インターネット上に構築に関する情報も豊富にあり、学習にも最適。 ※PostgreSQLを使う場合はLAPPと呼ぶこともある。
🔊 音声補足（池田先生）
サーバ関連の勉強をしてみたいなら、このLAMP構成から始めてみるのがおすすめ。 情報も豊富にある。← S05-A/B

第2章 まとめ

  • Webサーバはクライアント・サーバシステムの代表例
  • 多くのWebサービスはWebアプリケーションとして動作
  • 高レスポンス必須（3秒ルール）。サーバ負荷は流動的
  • 安定稼働・保守性向上のために3層アーキテクチャ（Web・AP・DB）という設計
  • LAMP構成（L・A・M・P）はすべてOSSで構成される代表的な構成
  • Web関連エンジニアはフロントエンド・バックエンド・インフラなど広い範囲に関連
CH 03

サーバOSとクライアントOSの違い

学習目標
  • サーバOSとクライアントOSの違いについて理解し説明できる

3-1 よく利用されるサーバOS / クライアントOS

区分主なOS
サーバOSLinux（RedHat系・Debian系）、Windows Server
クライアントOSWindows、macOS、iOS、Android

← S03:p.32

補足: Linuxにはクライアント向けのOSも存在するが、一般的にはサーバ用途で多く使われる。

用途の違い

項目サーバOSクライアントOS
対象企業・学校・データセンター等一般ユーザー
重視点構築効率・安定稼働・セキュリティ設定多機能性・操作性（マウス等）
メモリ上限大容量対応（Windows Server: Max 48TB）少ない（Windows 11 Pro: Max 6TB）
リモート管理充実（リモートアクセス対応）限定的

← S03:p.33, p.35

重要: 必ずサーバOSを使わなければならないという決まりはない。 ただし、使用するソフトウェアや機器の違いから、用途に合わせたOSを利用することが望ましい。
🔊 音声補足（池田先生）
クライアント向けOSはゲームや文書作成など多機能さが重視される。 対してサーバ向けOSはセットアップ直後に必要最低限のソフトウェアしか入っていない。 これにより不要なプログラムの起動を抑え、サーバ本来の処理にリソースを集中できる。 ← S05-A/B

3-3 サーバOSの種別：Windows系 vs UNIX系

🔊 音声補足（池田先生）
LinuxはいろいろなディストリビューションがあるためCLIが中心だが、 GUIがあるものも存在する。macOSやAndroidもUNIX系に属するというのは 意外に思う人もいるかもしれない。← S05-A/B

サーバ向けOSの特徴

特徴内容
① 構築作業の効率化 セットアップ直後は最低限のソフトウェアのみ。
→ セキュリティリスク低減・不要プログラムの起動抑制 → サーバ性能向上
② 稼働の安定性 多数クライアントからの接続要求に対応できるよう設計。
チーミング: 複数NICを組み合わせた冗長構成。片方断絶でも機能維持
③ パフォーマンスとリソース Windows Server: メモリ最大48TB（Windows 11 Pro は 6TB）

← S03:p.35

3-5 UNIX系OSがサーバ用途に向いている

Webサーバの利用統計（W3Techs調べ）: ← S03:p.36

OS系統スライド記載（2023年8月時点）最新（2025年12月時点）
Unix / Linux系 81.8% 約59%（Linuxのみ集計）
Windows 18.5% —
[外部補足] 統計の注意点（解決済）
スライド記載の81.8%は「Unix系OS全体（Linux + BSD + macOS等）」を集計したカテゴリで、現在のW3Techsの集計方法（Linuxのみ）とは集計対象が異なる。2025年12月時点でLinuxは全Webサイト中約59%（OS特定分）を占めており、スライド公開当時から傾向は同様。いずれにせよUNIX/Linux系が圧倒的多数を占めるという講義の主旨は変わらない。
出典: commandlinux.com（W3Techs集計ベース、2025年12月）← 外部補足・U01解消

LinuxがサーバOSとして選ばれる理由

  1. 長時間の連続稼働に強い
  2. 低スペックでも動作する → サーバ処理に多くのリソースを割ける
  3. アクセス管理の権限設定が厳密で安全性が高い

← S03:p.37

Linuxの成り立ち

項目内容
開発者リーナス・トーバルズ（Linus Torvalds）
開発年1991年（当時フィンランドの大学生）
ベースUNIXを参考にゼロから開発（UNIXは当初研究用として無料公開、後に有償化）
特徴オープンソース。改変・再頒布が認められる
派生様々なディストリビューション（Ubuntu・Rocky Linux・RHEL等）

← S03:p.37

🔊 音声補足（池田先生）
LinuxはUNIXを参考にゼロから作られた。UNIXは当初研究用として無料公開されていたが後に有償化。 それをベースに独自開発したのがLinux。オープンソース型なので改変も自由にできる。 様々なディストリビューション（派生版）があり、用途に合わせてカスタマイズされたものが インターネット上で公開されている。← S05-A/B

3-6 商用向けのLinux（RHEL）

項目無償ディストリビューションRHEL
費用無料（Ubuntu・Rocky Linux等）有償
サポートコミュニティフォーラム専用サポートあり
セキュリティアップデート通常タイミング無償版より先行して受けられる
商用向け専用機能なしあり

← S03:p.38

ポイント: 開発環境やセキュリティリスクの低いサービスであれば、 無償ディストリビューションで問題ない。

第3章 まとめ

  • サーバには必ずサーバOSを使わなければならないという決まりはないが、専用OSの方が向いている点が多い
  • Linuxがサーバで好まれる理由: 稼働安定性・リソース効率（低スペック動作）・権限管理の厳密さ
  • Linuxは基本的に無料。商用向けにはRHEL（有償）がある
  • Windows ServerはメモリMax 48TBまで対応（一般向けの6TBより大幅に多い）
CH 04

オープンソースのシステム

学習目標
  • オープンソースのソフトウェアについて理解し説明できる
  • オープンソースのソフトウェアを利用したシステムについて理解し説明できる

4-1 なぜサーバシステムでOSSを取り上げるか

Webサーバで使用するソフトウェアには無償のものが多く、 サーバ向けOSとしてLinuxが多く用いられるが、これもOSS。 サーバシステムに利用するソフトウェアはオープンソース化されているものが多い。 ← S03:p.43

4-2 オープンソースとフリーソフトの違い

項目オープンソース（OSS）フリーソフト
無料利用○○
ソースコード公開○（公開）✕（非公開が多い）
改変・再頒布○（条件付きで可）✕（不可が多い）
ライセンス種類GPL・MIT・Apache等独自条件が多い

← S03:p.44

🔊 音声補足（池田先生）
よく混同されるが、OSSはプログラムのコード構造まで全部公開されている点が フリーソフトとの大きな違い。ライセンス形態も複数あり、条件が異なる。 ← S05-A/B

オープンソースの定義（10項目）← S03:p.45

  1. 再配布を自由に認めること
  2. ソースコードを無償で配布すること
  3. 派生物のソフトウェアには配布を許可すること（同ライセンス適用）
  4. ソースコードが作者オリジナルかわかるようにすること（コードの完全性）
  5. 個人やグループに対する差別をしないこと
  6. 使用分野に対する差別をしないこと
  7. プログラムに付随する権利は全再頒布者に平等に与えられること
  8. 特定の製品に限定したライセンスにしないこと
  9. 他のソフトウェアを制限するライセンスにしないこと
  10. 技術的に中立であること

出典: Open Source Initiative「The Open Source Definition」https://opensource.org/osd/

4-4 OSSのメリット・デメリット

種別内容
✓ メリット
頻繁な更新		ソースコード公開でバグ発見が早く開発スピードが速い
✓ メリット
自由に改変		条件付きでソフトウェアの改変が可能
✓ メリット
低コスト		自前のコーディング・テスト時間を削減
✓ メリット
透明性・安全性		ソースコード公開で怪しいものが流通しにくい
✓ メリット
ベンダーロックイン対策		汎用製品の組み合わせで特定企業への依存を防げる
✗ デメリット
保証なし		不具合・脆弱性に対する保証がない
✗ デメリット
開発継続の不確実性		コミュニティの状態に依存。人気が落ちると更新されなくなる可能性

← S03:p.46

🔊 音声補足（池田先生）
ただし「保証がない・開発が続くか不明」はOSSに限った話ではない。 業者にシステムを作ってもらっても、その業者が永続的にメンテナンスできるとは限らない。 ベンダーロックインの問題を考えると、OSSを使う方がむしろ健全な場合もある。 ← S05-A/B

代表的なOSSの例← S03:p.47

カテゴリ代表的なOSS
OSUbuntu、Rocky Linux 等
プログラム言語PHP、Ruby、Python 等
アプリケーションFirefox（ブラウザ）、Thunderbird（メール）、LibreOffice（オフィス系）等

4-6 OSSを利用したシステムの例：LAMP + WordPress

Webサーバ構成（LAMP + WordPress）: OS: Linux（Ubuntu） ├ Apache ← Webサーバ ├ PHP ← プログラム言語（PHPを使う場合APサーバ不要） └ MySQL ← データベース ↓ CMS: WordPress をインストール・設定

← S03:p.50

活用場面:

  • HTML・CSS・PHPの技術を学ぶ環境としても利用できる
  • ※XAMPPなど、より簡単にインストールできる環境もある
🔊 音声補足（池田先生）
LAMP構成まで作れるようになったら、CMSと組み合わせてちょっとしたWebサーバを 自分で作れる。WordPressはHTML・CSSのデザイン変更の学習にも使える。 フロントエンドエンジニア関連技術の学習環境構築にも有用。 ← S05-A/B

第4章・第7回 まとめ

  • OSSとフリーソフトはいずれも無償だが、OSSはソースコードも公開・改変可能
  • OSSのメリット: 頻繁な更新・低コスト。デメリット: 保証なし・開発継続不確実
  • Web関連システムではOSSの組み合わせで構築することが珍しくない
  • サーバシステムの基礎: クライアント・サーバシステムが基本。ネットワーク知識も重要
  • Webサーバ: 高レスポンス（3秒ルール）・3層アーキテクチャ・LAMP構成
  • サーバOS: Linuxが好まれる（安定性・リソース効率）。RHEL（商用有償）もある

📚 補足B: 知っておくべき背景知識

LAMPのPはPHPだけでなくPerlやPythonも含まれる。PostgreSQLを使う場合はLAPPと呼ぶこともある。← S03:p.27
VPSとクラウドの関係: VPSは仮想化技術で専用サーバを仮想的に再現するもの。クラウド（AWS・GCP・Azure）はより大規模な仮想化・マネージドサービスを提供。仮想化技術は第8回で詳しく扱う。← S03:p.15
OSSライセンスの代表例: GPL（改変版も同ライセンスで公開義務）・MIT（商用利用・改変・再配布が自由）・Apache License（特許権の明示的許諾）。← S03:p.44
[外部補足] Windows Server メモリ上限 48TB は確認済み: TechTarget、Microsoft Learn 両方で「Standard/Datacenterとも最大48TB RAM対応」と記載されている。Windows 11 Proは最大6TB。スライド記載の数値は正確。出典: TechTarget「Compare the features in the Windows Server 2022 editions」
[外部補足] Linuxのサーバ市場規模（2025年最新）: Linuxはサーバ用OSの約44.8%のシェアを持ち（2024年）、クラウドワークロードの約49.2%（2025年Q2）を占める。AWS・GCP・Azure上のVMの約92%がLinuxで動作。TOP500スーパーコンピューターは100%Linuxで稼働（2024年11月時点）。出典: commandlinux.com（2025年）
[外部補足] WebサーバはNginxがApacheを逆転: かつてApacheがWebサーバの王者だったが、2023年頃にNginxが市場シェアで逆転。2025年12月時点でNginxは約33%のWebサーバシェアを持ち、Apache（約25%）を上回る（W3Techs集計）。NginxはイベントドリブンなノンブロッキングI/Oアーキテクチャで大量同時接続に強い。ApacheはモジュールによるカスタマイズやPHPとの連携が容易。多くの本番環境では「Nginxがリバースプロキシとして前段に立ち、動的コンテンツはApacheやPHPに渡す」というハイブリッド構成も普及している。出典: commandlinux.com（W3Techs集計ベース、2025年12月）

🎯 補足C: 試験・レポートで重要な観点

クライアント・サーバシステムの定義と動作原理（サーバが自発的に動かない点）

3層アーキテクチャの3つの層と各役割（Web=窓口・AP=処理中枢・DB=データ格納管理）
LAMP構成の各文字が何を指すか（Linux・Apache・MySQL・PHP/Perl/Python）
OSSとフリーソフトの違い（ソースコード公開・改変可否）
OSSのメリット5点・デメリット2点がきちんと答えられること
LinuxがサーバOSとして好まれる理由3点（連続稼働・低スペック動作・権限管理の厳密さ）
RHELと無償Linuxの違い（サポート・先行セキュリティアップデート・商用機能）
