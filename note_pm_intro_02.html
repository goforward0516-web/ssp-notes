プロジェクトの3要素

定義

プロジェクトとは「目的が決まっている」「資源が決まっている」「期限が決まっている」という3つの要素を持つ活動である。

🎙 音声補足

この3つの要素はプロジェクトマネジメント入門の授業全体を通じて最も重要な概念であり、必ず覚えておく必要がある。目的・資源・期限のいずれか一つでも不明確な場合、プロジェクトとして構成することができない。← S05-A

プロジェクトマネジメントの定義

定義

プロジェクトをどのように遂行するか計画を行い、その計画に基づき実行し、実行の過程を測定しながら、プロジェクトの目的を達成できるようにコントロールしていくこと。← S03:p.6

🎙 音声補足

「コントロール」とはうまくいかなかった部分を是正することを意味する。うまくいっている部分には手を打たなくてよい。計画→実行→測定→コントロールの流れが重要。← S05-A

ソフトウェア規模と失敗率

🎙 音声補足

Capers Jones「ソフトウェアの成功と失敗」より、ソフトウェアの機能数（FP: Function Point）が増えるほど、中止・遅延プロジェクトの割合が増加することが示されている。規模が大きくなるほど被害も大きくなるため、プロジェクトのコントロールが不可欠となる。← S03:p.7 / S05-A

プロジェクトマネジメントの2つの課題

  • 相反する要求事項の最適なバランスをとること — スケジュール・コスト・品質・人・技術・目的が相互にトレードオフの関係にある
  • リスクを顕在化し、リスクを減らすこと — どのようなリスクがあるかを明示化し、その対処を行う
🎙 音声補足

例えば、スケジュールを延ばすとコストが増加し、品質を上げようとするとコストも増える。これらの相互関係を理解した上でバランスを取ることがマネジメントに要求される。← S05-A

§ 1 第1章 プロジェクトを取り巻く課題

学習目標: プロジェクトを取り巻く課題について理解し、説明できる。← S03:p.12

1-1 複雑化する私たちの環境

現代のプロジェクト環境は以下の4つの方向で複雑化が進んでいる。← S03:p.13

具体例

領域複雑化の具体内容
夏休みの宿題塾の宿題の増加、パソコン利用、英語科目の追加、デバイスの多様化
家の建築建築基準法の制限増加、良質な原材料の減少、新工法（床暖・オール電化等）の選択肢増加
コンピュータシステム規模の拡大、海外ユーザーとの連携増加、セキュリティ・通信速度・OS対応の複雑化
🎙 音声補足

大規模開発の例として、300名規模のシステム開発現場では、複数社の共同開発・オフショア/ニアショア（海外開発チーム）の参加・複数の開発環境・多数のユーザー・テレワーク開発が組み合わさる。こうした複雑な環境では、しっかりとしたマネジメントがないとうまく連携が取れない。← S05-A

1-2 幻想の標準化

「時代が変わっても楽にはならない。むしろ…」 ← S03:p.17

🎙 音声補足

時間が経てばシステム開発が楽になると思いがちだが、実際は標準化によって一時的に安定期が訪れても、その期間は短い。次の環境変化が生じ、また混沌とした状態に戻る。標準化自体も進化し続けなければならず、さらなる複雑化が続く。← S05-A

第1章まとめ

  • 私たちが取り組むプロジェクト（夏休みの宿題・家の建築・料理・システム開発）はますます複雑な事情の中で解決しなければならない
  • そのためには成功確率を上げるための作法である「プロジェクトマネジメント」を学び工夫していく必要がある ← S03:p.18

§ 2 第2章 プロジェクトのライフサイクル

学習目標: プロジェクトのライフサイクルについて理解し、説明できる。← S03:p.21

2-1 ライフサイクルの定義と概念

定義 — ライフサイクル

プロジェクトが始まってから終わるまでの一連の流れ。「サイクル」は繰り返しがあることを意味する。← S03:p.23

🎙 音声補足

プロジェクトの3要素のうち「期限が決まっている」ということは、始まりがあり終わりもあることを意味する。家の建築で例えると「建築→補修→補修→補修→取り壊し」という繰り返しのサイクルが存在する。← S05-A

プロジェクトの起承転結フェーズ

講師は100以上のプロジェクトを経験してきたが、必ずプロジェクトには起承転結があると述べている。← S05-A

ライフサイクルの具体例

例ライフサイクルの流れ
家づくり 計画 → 契約 → 設計仕様書・設計図面作成 → 基礎工事完了 → 棟上げ → 施主検査 → 引き渡し（完成）← S03:p.25
システム構築 構築 → 利用・運用 → 変更 → 利用・運用 → 変更 → 再構築（大規模刷新）← S03:p.26

2-2 プロジェクトの始まり — 宣言の重要性

🚀 プロジェクトの始まりは「始めます」と宣言した時。

始まるまではなんとなく話し合ってもよいが、始まる時には周囲からわかるように宣言する必要がある。← S03:p.27

🎙 音声補足

「システムを作る人たちだけでなく、周囲から見てもあのプロジェクトがこういう目的でスタートしたとわかるようにすることが重要」というのがポイント。単に内部で決めただけでは不十分で、正式に宣言することで周囲も認識する。← S05-A

プロジェクト憲章（宣言の文書例）

📄 プロジェクト憲章とは

プロジェクトの開始を公式に認めると書いた文書。目的・期間・リーダー・場所・組織などが記載されており、周囲の人から見てプロジェクトが立ち上がったことがわかるようにする文書。← S03:p.27

始まりの例

例スタートとなる行動
料理「献立を決めた時」— 心の中で決めた、または献立を書いた時← S03:p.28
家を建てる「家族会議で決まった」または「銀行でローンを組む」← S03:p.29
システム開発利用部門と情報システム部門がシステム企画書を作成し、経営会議で承認された時← S03:p.30
🎙 音声補足

「こんなシステムがあったらいいね」と話しているだけではプロジェクトはスタートしていない。企業の場合、企画書を作成しただけでなく経営会議で承認されることで初めてプロジェクトがスタートする。「立てたいね」と思っているだけでも同様にスタートとはならない。← S05-A

2-3 プロジェクトの終わり — 宣言と次への接続

🏁 プロジェクトの終わりも「終わります」と宣言する必要がある。

終わる時にも周囲からわかるように宣言する。この宣言がないとずるずるとプロジェクトが進行してしまう。← S03:p.31

📋 プロジェクト最終判定会

プロジェクトの終了を公式に認める会議。当初の目的が達成されたかを判定し、終了を宣言する。目的が達成されていない場合でも終了（中止）の宣言をすることがある。← S03:p.31

🎙 音声補足

宣言をしないとずるずるとプロジェクトが進行してしまう。一旦区切りをつける意味でも、終わりの宣言は非常に重要。目的未達成でも中止の宣言をすることがある。← S05-A

終わりの例

例終わりとなる時点
料理献立通りの料理が完成した時、または食事・食器片付けが終わった時← S03:p.32
家を建てる家が完成し施工業者から施主へ引き渡された時、または引っ越して住み始めた時← S03:p.33
システム開発システムが本番稼働（カットオーバー）した時（長いスパンでは最初の月末処理・年末処理確認後も）← S03:p.34
🔄 終わりは次の始まり

あるプロジェクトの終わりは、次のプロジェクトや次の運用の始まりでもある。ライフサイクル全体を考えて活動することが重要。「保守がしやすいか」「追加のコストはどうか」という観点でライフサイクル全体を見ることが求められる。← S03:p.35

🎙 音声補足

システムが本番稼働すると運用が始まり、不具合が出てきたり変化が生じたりする。それを改善するという次のプロジェクトの始まりでもある。長い目で見たプロジェクトを考えることが重要。← S05-A

第2章まとめ

  1. プロジェクトには長い視点・短い視点のプロジェクトがある
  2. どちらにも始まって終わるというライフサイクルがある
  3. プロジェクトの始まりと終わりは宣言しなくてはならない
  4. 一つのプロジェクトの終わりは次のプロジェクトの始まりでもある← S03:p.36

§ 3 第3章 プロジェクト型組織の形態

学習目標: プロジェクト型組織の形態について理解し、説明できる。← S03:p.39

3-1 組織の定義

定義 — 組織

ある共通の目的を達成するために、2人以上の人が集まって、思いを合わせて、一定のルールの中で活動する仕組み。← S03:p.40

🎙 音声補足

組織の重要点：①共通の目的があること、②それを全員が認識していること、③達成したいと思っていること。単に100人を集めただけでは組織として機能しない。「組織にしなくては、ただ人数がたくさんいてもプロジェクトは実行できない」。← S05-A

なぜ複数人が必要か

1人でできる仕事は限られているため、複数人での組織が必要になる。プロジェクトの規模が大きくなるほど関係する人数が増える。← S03:p.41

規模例必要な人員
犬小屋を作る1〜2人
一軒家を作る10〜20人程度
駅のホームをリニューアルする何千〜何万人

組織形態の種類

組織形態特徴説明
階層型組織 オーソドックスな伝統的手法 役割によってグループが層のようになっている。上→下への指示と下→上への報告の繰り返し。← S03:p.47
クロスファンクショナル型組織 近年多用 機能（ファンクション）が縦横にクロスして任せられながら仕事をする。詳細は第9回で学習。← S03:p.46

3-2 階層型組織の構造と特徴

階層型組織の特徴

  • 上の層の人がすぐ下の層の人に指示をする
  • 下の層の人は指示されたことを実行し、状況を上の人に報告する
  • この構造が横に増えても同じ仕組みで展開できる（スケーラビリティがある）
  • 典型例: 軍隊（上の命令に必ず従い、下は実行して報告する）← S03:p.49
🎙 音声補足

料理の例：総料理長の下に料理シェフとデザートリーダーがいて、その下にメンバーがいる構造。200名のディナーパーティーなどの大人数を対象にした場合、こうした階層型の組織が有効。メンバーがそれぞれの役割（「私はデザートを担当する」）を明確に持ち、指示-実行-報告の流れを守る。← S05-A

3-3 メンバー集めのポイント

💪 ベストメンバーで行こう！

ポイント内容
悔いのない人選 後で「あの人を入れておけば」と後悔しないために、この分野では一番だと思った人を熱意を込めて説得して連れてくる← S03:p.51
専任できる体制 名前だけプロジェクトに入っていて実際には別の人が手伝うという形はダメ。ベストメンバーが実際にプロジェクト期間中に専任して作業することが必要← S03:p.51
日頃の努力 他部門・他社との日頃からのコミュニケーションと関係構築が重要。「この人のプロジェクトなら出す価値がある」という関係性を普段から作っておく← S03:p.51
🎙 音声補足

日頃から良い関係を築いておかないと、いざベストメンバーを要請したときに「うちは重要な人材なので出せません」と断られてしまう。最初の組織形成は非常に重要な仕事で、プロジェクトのパフォーマンスを決める。← S05-A

第3章まとめ

  1. プロジェクトを複数人数で行う際には組織が必要になる
  2. 典型的な組織形態として階層型組織とクロスファンクショナル型組織がある（この回では階層型を解説）
  3. メンバー集めはベストメンバーで悔いのない人選を行うことが重要← S03:p.52

§ 4 第4章 プロジェクト運営のしくみ

学習目標: プロジェクト運営のしくみについて理解し、説明できる。← S03:p.55

4-1 運営の難しさと3つのポイント

🎙 音声補足

プロジェクトの規模が大きくなり関係する人数が増えるほど、コミュニケーション・イメージ伝達・共有の難しさが増す。1人で行う場合は問題ないが、3人・40人（クラス）・100〜200人（町内）と増えるにつれ、意識の共通化が難しくなる。← S05-A

組織としての運営の3つのポイント ← S03:p.57

#ポイント良い例悪い例
① 明確な役割と責任 デザート担当の料理人・内装担当の工務店員・機材準備担当者など、誰が何をするかが全員に明示されている みんなが同じところに固まる（役割分担なし）← S03:p.58
② 状況の正確な把握と共有 料理の進捗表・工事の日程表（更新版）・システム開発スケジュール（更新版）を全員が見られる 書いたものが古く、噂が蔓延し実態がわからない← S03:p.59
③ 会議などの公式な意思決定 献立決定会議・工事内容決定会議・システム開発月例報告会議など いつの間にか誰かの判断で方向が変わっている← S03:p.60

4-2 主要な会議体 ← S03:p.61

① ステアリング・コミッティ

役割: プロジェクトの方向性を決定する責任者の会議

語源: ステアリング（車のハンドル・船の舵）

メンバー: 各グループの責任者

頻度: 方向転換が必要な時

Go / Return / Stop の判断はここで行う

② 定例進捗報告会

役割: 問題・予実績を定例的に報告する

内容: 問題報告・予実績報告

発表者: プロジェクトメンバー

頻度: 月1回〜週1回〜毎日

月例会 / 週例会 / 朝礼・終礼 / 9時から会議

③ マイルストーン・レビュー

役割: 節目での評価・判定

内容: 品質・コスト・スケジュールの実施状況

報告者: プロジェクト責任者（PM）

受け側: ステアリング・コミッティ

結論: Go / Return / Stop

ステアリング・コミッティの具体例 ← S03:p.63

例参加者決定する内容
料理各セクション（料理・前菜・デザート・ドリンク）の長今回はこれでいこう、状況が変わったのでこう変えよう
家作り住まう家族と建築の責任者この家をこうしていこう、方針を確認・変更
システム開発開発依頼責任者と開発責任者これでやっていこう、状況が変わったのでこう変えよう

マイルストーン・レビューの判定 ← S03:p.66-67

判定意味
Go（合格）このまま進めてよい
Return（やり直し）修正して再度進める
Stop（中止）このプロジェクトを中止する。問題が解決されないままシステム開発が進むと使えないシステムが完成してしまうため、この判断が非常に重要。← S05-A
🎙 音声補足

ストップをかけないとそのまま赤字が垂れ流されたり、問題が解決されないままシステム開発が進んで最終的に使えないシステムが完成するという事態になりかねない。この判断は非常に重要。← S05-A

共有・報告の方法 ← S03:p.68

関係者に日頃から関心を持ってもらい、状況を理解して早く手を打てるようにするため、以下のようなコミュニケーションツールを活用する。

  • プロジェクト・メルマガ
  • プロジェクト・ブログ
  • メッセンジャーアプリ

第4章まとめ ← S03:p.69

  • プロジェクトが大きくなるほど、関連する組織や人が多くなるので、組織としての運営を効率よく行うことが大切
  • 運営のポイントは明確な役割と責任・状況の正確な把握と共有・会議などの公式な意思決定の3つ

プロジェクト憲章・スコープ記述書・PM計画書の3段階

📄 外部補足
文書名タイミング記載内容
プロジェクト憲章 立ち上げ時 何のプロジェクトか・目的・期間・リーダー・予算概要。「このプロジェクトが存在する」ことの公式認証文書。PMBOKの立ち上げプロセスの主要成果物。
プロジェクトスコープ記述書 計画フェーズ初期 プロジェクトの範囲（何をやるか・何をやらないか）を詳細に定義。スコープクリープ（無制限な範囲拡大）を防ぐ基準文書。
プロジェクトマネジメント計画書 計画フェーズ スケジュール・コスト・品質・リスク・コミュニケーション等の全管理計画を統合した文書。PMBOKでは10の知識エリアそれぞれのマネジメント計画を含む。

ステアリング・コミッティの実務的な位置づけ

💼 外部補足 — IT用語辞典e-Words / Promapedia 参照

授業の定義（「異なる目的を持つグループ間で合意を得るための委員会」）に加え、実務上の特徴：

  • PMが対応できない上位課題（部門間対立・大幅なスコープ変更・大規模予算調整）の意思決定を行う場
  • 定例ミーティングとの違い：定例は「進捗確認」が目的、ステコミは「方向性の確定と重要課題の解決」が目的
  • 参加者はプロジェクトオーナー・経営層・主要部門責任者など意思決定権を持つ人物が原則
  • 開催タイミングは事前にプロジェクト計画書で明確化しておく（「必要に応じて」では遅延しがち）
  • 「ステコミ」「SC（エスシー）」と略されることが多い

起承転結とPMBOK 5プロセス群の対応

📖 外部補足 — PMBOK第6版
授業の「起承転結」PMBOKプロセス群主な活動
起（初期フェーズ）立ち上げプロジェクト憲章作成・ステークホルダー特定
承（中間フェーズI）計画→実行スコープ記述書・PM計画書作成→実作業開始
転（中間フェーズII）監視・コントロール進捗測定・是正処置・変更管理
結（最終フェーズ）終結成果物引き渡し・プロジェクト終了宣言

※監視・コントロールは計画フェーズ開始と同時に並行して続き、終結まで続く点が特徴。PMBOK第7版（2021年）では5プロセス群から12の原理・原則に改定されたが、基本的な流れの概念は引き継がれている。

🎯 補足C: 試験・レポートで重要な観点

  • プロジェクトの3要素（目的・資源・期限）のそれぞれが欠けると何が問題になるかを具体例で説明できること
  • 「宣言が必要」な理由：始まり（=プロジェクト憲章）・終わり（=最終判定会）の宣言がないとどうなるかを説明できること。「宣言しないとずるずると進行する」は重要なキーワード。
  • ライフサイクルの「終わりは始まり」：一つのプロジェクトが終わると次の運用・改善プロジェクトが始まる連鎖を説明できること。「保守しやすいか」「追加コストはどうか」という長期視点が重要。
  • 組織の4要素定義：「共通の目的を達成するために、2人以上が集まって、思いを合わせて、一定のルールの中で活動する仕組み」を正確に言えること。「100人集めても組織にならない」理由を説明できること。
  • 階層型組織の指示-報告サイクル：「上の層→下の層への指示」「下の層→上の層への報告」の繰り返し構造、および横展開できるスケーラビリティを説明できること。
  • 運営の3ポイント：①明確な役割と責任・②状況の正確な把握と共有・③公式な意思決定——それぞれの「悪い例」（役割が固まる・情報が古い・いつの間にか方向が変わる）とセットで説明できること。
  • 3つの会議体の区別：ステアリング・コミッティ（方向性・経営層）/ 定例進捗報告会（定期・メンバー同士）/ マイルストーン・レビュー（節目・PMがステコミに報告）の違いと用途を区別できること。
  • Go / Return / Stop の判定：マイルストーン・レビューの3判定の意味。Stop判定が重要な理由（問題を抱えたまま進むと使えないシステムが完成する）を説明できること。

ウォーターフォール型 vs アジャイル型

💡 外部補足

この授業では主にウォーターフォール型（要件定義→設計→開発→テスト→リリースを順番に進む）の管理を扱う。一方、現代ではアジャイル型（短い反復サイクル「スプリント」を繰り返し価値を継続提供する）が広く普及している。

ウォーターフォール型アジャイル型
計画プロジェクト開始時に全体計画スプリントごとに計画
変更対応変更は高コスト変更を歓迎
組織形態階層型が多いクロスファンクショナル型が主
向いているプロジェクト要件が明確・安定しているもの要件が変化しやすいもの

クロスファンクショナル型組織の詳細と、アジャイルとの関係は第9回「プロジェクトの組織マネジメント」で扱われる。

PMPとPMI資格

🏆 外部補足

PMP（Project Management Professional）はPMIが認定するプロジェクトマネジメントの国際資格。PMBOKの知識を実務に応用できることを証明する。プロジェクトマネジメントのキャリアを目指す場合、この授業の学習内容は資格取得の基礎となる。
