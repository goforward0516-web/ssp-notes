第1章

管理するには、まず測る

なぜ測ることが必要か

同じようなことに何度トライしても良い結果が出ない原因は、「目標が悪い」か「やり方が悪い」かのいずれかである。結果を改善するには、どこが悪いのかを特定して改善していく必要がある。そのための第一歩が「測る」ことである。← S03:p.14

🎙 音声補足
プロジェクトは関係者で協力して進めるものである。最も難しいのは「人に動いてもらうこと」。自分が何をすべきかは分かっていても、他者に動いてもらうには状況を正確に伝えることが必要である。そのとき数値による説明は言葉による説明より格段に分かりやすく、人が動きやすくなる。← S05-A
数値で測れないものは、管理できない。
管理できないものは、制御できない。
制御できないものは、なされるがままの状態を招く。

← S03:p.16

🎙 音声補足 — 数値管理の具体例
曖昧な目標の例：「秋までにダイエットを頑張る」
→ 「いつの秋か」「どれくらいやったら頑張ったことになるか」が不明。

数値で管理した例：「20XX年10月15日 9:00 PMに体重が60kg以下であること」
→ いつまでに・何が・どの水準か、が明確になる。本来はシャープな数値（60kg以下ではなく60kgや59kg）が望ましい。← S03:p.17 / S05-A

ものさし（計測指標）の選び方

数値で測るときに「何で測るか」というものさしの選択が非常に重要である。実際にやってみると難しいため、測れるものを徹底的に探すことが求められる。← S03:p.18

優先度ものさしの例（システム開発）測れるか補足
① 高支出した費用：支払い金額◎ 明確実際の支払いが存在するため測りやすい
② 中プログラムの出来高：完成したモジュール数・ステップ数・工数◯ 可能カウントで計測できる
③ 低仕様書の出来具合：書いた枚数・合格した枚数△ 限界あり枚数が多くても内容の質を反映しない。代替手段がない場合のみ使用

← S03:p.18 / S05-A

[外部補足]
ものさし（KPI: Key Performance Indicator）の設計は、数値で追いやすく・目標と連動し・偽装しにくいものを選ぶことが原則。「書いた枚数」のような代理指標（Proxy）は、本来の目標（品質）を反映しないため「グッドハートの法則」（測定対象になった指標は良い指標でなくなる）の例として挙げられることが多い。

評価とは、問題を発見するために、「達成できているか・できていないか」を明確にし、現状を判断することである。評価のためには、計画を立てたときと同じものさしで実績を追うことが必要。← S03:p.19

問題の認識

【核心原則】

問題を認識した時点で、その問題の8割は解決している。
問題は見えにくいため、数値で測り「計画値と実績値のギャップ」として可視化することが問題解決の鍵となる。← S03:p.20

🎙 音声補足
「真の問題が見えにくい」ことが最大の障壁である。数値で測ることで「計画値と実績値のギャップ」として問題が可視化され、そのギャップに対して対策を打てる。いかに問題の本質を認識するかが重要。← S05-A

第1章まとめ← S03:p.21

  • 活動の状況把握は、数字で測れるものにする
  • 管理するには、数値で計画と実績を追っていく
  • 問題を認識した時点でその問題の8割は解決している
  • いかに問題を認識するかが、問題解決の鍵
第2章

原因を探って改善する

インプット・プロセス・アウトプット

【定義】プロジェクトのプロセス

プロジェクトの中のプロセスとは、アウトプット（成果物）を得るための活動のことである。良いアウトプットを得るには、良いインプットと良いプロセスの両方が必要。← S03:p.25

例インプットプロセスアウトプット
料理材料・調理器具・料理人調理料理（完成品）
家作り設計図・資材・大工さん・土地建築家

← S03:p.26-27

アウトプットが良くない場合は、インプットかプロセスのいずれか（または両方）に問題（原因）がある。← S03:p.28

🎙 音声補足 — 結果が思わしくない時の原因分類
  • ① 予想外の出来事が起こった → 実は予想が足りていなかった可能性がある
  • ② やり方が良くなかった → 対策が打てる。どこを改善するかを特定する
  • ③ そもそも計画が間違っていた → 次回以降の計画立案の改善テーマになる
← S03:p.29 / S05-A

問題の原因を探るには、発生した現象は氷山の一角と認識することが重要。表面の現象に対して「真の原因を正しく認識する」ことで初めて正しい対処ができる。← S03:p.30-31

🎙 音声補足 — ナンがベチャっとしている例
ナン（インドのパンの一種）がベチャっとした原因として考えられるもの：焼き方の間違い・材料が違う・生地に水分が多すぎ・焼くときの厚みがありすぎ・発酵が足りないなど。原因を正しく認識しないと、違う対策を打つことになり結果が改善されない。← S05-A / S03:p.30

分析の手法：QC（品質管理）の七つ道具（パレート図・特性要因図・グラフ管理図・チェックシート・散布図・ヒストグラム・層別）は第8回で詳しく学ぶ。今回はQCの七つ道具に含まれないKJ法を扱う。← S03:p.35

KJ法を使った分析

【定義】KJ法

カードを作り、それをまとめていく過程で各カード（現象）が何を語っているかを紡ぎ出す分析手法。川喜田二郎（かわきた じろう）氏が考案したため、氏のイニシャルを取って「KJ法」と呼ぶ。← S03:p.36

①
要素を書く
思いつく要素をカードにできるだけたくさん書く。複数人で行うと多くの視点が集まる
②
グループ化
キーワードや似たエリアのものでカードをまとめてグルーピングする
③
ラベル付け
各まとまりに名前（ラベル）をつける。まとまりが「何を物語っているか」を表す言葉を選ぶ
④
文章化
図解した全体を眺め、「○○とは〜〜のことである」と文章化する

← S03:p.37-40

🎙 音声補足 — 「よい家とは？」の適用例
①カードに記載：「鍵が3重である」「風呂が広い」「床暖房がある」「玄関が広い」「木の香りがする」「団らん空間がある」
②グルーピング：「風呂が広い・玄関が広い」→広さ系、「鍵が3重・床暖房がある」→設備系
③ラベル：「広さが確保されている」「機能的である」
④文章化：「よい家とは、広さが確保され、機能的な家のことである」（あくまで例題）← S05-A / S03:p.37-40

KJ法は、プロジェクト実行中だけでなく、システムを使う背景の観察など状況調査にも活用できる。← S05-A

不具合の原因は何であるかと課題の本質を見極めて対処することが重要。
真の原因を対処しないと、違うところに不具合が出る・長期的な対処にならない。

← S03:p.41

なぜを5回繰り返す（トヨタ式）

【定義】なぜを5回繰り返す

トヨタ式（トヨタ生産方式）で改善を行う際の標語。表面的な原因にとどまらず、根本原因（真の原因）にたどり着くために「なぜ？」を連鎖的に5回問い続ける手法。← S03:p.42

🎙 音声補足 — システム開発の適用例（5回の「なぜ」）
  1. 現象：システムが途中で止まった
  2. なぜ1：例外処理のところに抜けがあったから
  3. なぜ2：発生パターンの整理をして「その他」パターンを加えなかったから
  4. なぜ3：条件分岐のルールを作っていなかったから
  5. なぜ4：各プログラマーが処理パターンを認識していると思っていたから
  6. なぜ5：各プログラマーに任せても品質は保たれると思い込んでいたから
真の解決策：例外処理を直すだけでなく、プロジェクト内のプログラミングルールを整備する（条件分岐時の必須ルール・データベース操作ルールなど）。ルールがあることでどんなプログラマーが参加しても一定の品質が保たれる。← S03:p.42 / S05-A

第2章まとめ← S03:p.43

  • プロジェクトの途中で結果が思わしくないときは、原因を科学的に探り速やかに対応する
  • 問題を表面的でなく、真の原因を探り手を打つ
第3章

PDCAとは何か？

PDCAの概要

【定義】PDCA

Plan（計画）・Do（実行）・Check（評価）・Action（改善）の頭文字を取ったもの。計画したことを達成するための仕事の作法（サイクル）。ゴールを意識して回し続けることで活動レベルを改善していく。← S03:p.47

🎙 音声補足
本講義では動詞で揃えると「Act」となるが、「アクションを取る」という意味でAを「Action」として採用する。PDCAは同じことの繰り返しが行われる場合・あるゴールの達成に向けて同じことを繰り返せる際に特に有効。← S03:p.47 / S05-A

PDCAの各ステップと役割

ステップ役割重要ポイント（G-PDCAの課題）
Goal 何のために・何をするのか 真のテーマであるか。具体的で数値化されているか。ゴールがブレると以降の全てがブレる
Plan 目標を設定し、それを実現するためのプロセスを設計（改訂）する 実現可能な計画であるか。与えられた期間・資源でできるか
Do 計画を実施し、そのパフォーマンスを測定する 計画に沿って実行する。途中の変更と経過記録が重要。事前に決めた指標を計測・記録する
Check 測定結果を評価し、結果と目標を比較・分析する 淡々と差異比較を行う。計画値と実測値の差分とその原因を確認する
Action プロセスの改善・向上に必要な変更点を明らかにする 次回に有効な改善計画を立てる。「こうしたらいいかも」という曖昧なものではなく、次に実際に取れる改善計画を立てる

← S03:p.50-53,58 / S05-A

スパイラルアップ

【定義】スパイラルアップ

PDCAは一回で終わりでなく、Actionの結果をさらに次のPlanに結び付け、らせん階段のように次々と上に上がっていくことをスパイラルアップという。（スパイラル＝らせん）← S03:p.55

🎙 音声補足 — PDCA事例
旅行の例：金沢への2泊3日旅行を計画→実行→「祭りの日程に合わせるべきだった」と評価→改善→来年の旅行計画に反映

システム開発の例：「システムのレスポンスを3秒以内にしよう」と計画→システムを作る→ユーザー環境で計測→プログラムのチューンアップ（改善）→プログラムを修正して再度3秒に挑戦

料理の例：チキンカレーを作ろう（計画）→作る（実行）→食べてみて評価→「辛みをココナツミルクで調整」（改善）→ジア風チキンカレーを作る（次の計画）← S03:p.48-49,54 / S05-A

最も重要なゴール設定の5条件

G-PDCAにおいて最も重要なのはゴール設定である。ゴールがブレると、その後のプランが全てブレる。← S03:p.59

①
組織にとって真に解決すべき課題であること
②
具体的であること（できれば数値で示される）
③
達成までの期間が決まっていること（中長期と短期に分割も可）
④
資源の確保が約束されていること
⑤
組織内で共有されていること

← S03:p.59 / S05-A

🎙 音声補足
ゴールが具体的でないと「達成できたかどうかの判断」がつかない。成功か失敗かが判別できなければ、PDCAを有効に回すことができない。⑤の組織内共有がないと、一部の人だけが知っているゴールとなりチームとして進められない。← S05-A

PDCAとOODA — 2つのループの使い分け

【定義】OODA（ウーダ）

Observe（観察）→ Orient（方向づけ）→ Decide（意思決定）→ Act（行動）の頭文字を取ったループ。変化が激しい社会で、これまで行ったことのない挑戦を行う際に有効。← S03:p.60

PDCA
  • 過去の経験値が生きる場合に有効
  • 繰り返し同じことを行う際に有効
  • 計画に時間をかけて精度を高める
  • ゴールに向けて着実に品質を上げる
OODA
  • 過去の経験が活きない + 迅速な実行が必要な場合
  • 変化が激しく先が見えない場合
  • 問題を先延ばしせず即対応できる
  • スピード感ある活動が可能

← S03:p.60 / S05-A

🎙 音声補足 — OODAの釣りの例
初めての釣りポイントで（経験が活かせない状況）：
① 状況を観察する（Observe）→ ② 「アジなどの小魚がいそうだ」と方向付け（Orient）→ ③ 「小さめの仕掛けで餌釣りしよう」と意思決定（Decide）→ ④ 実際に釣ってみる（Act）→ ① 反応を観察する（次のObserveへ）
反応がなければ方向付けを変え、別の仕掛け（ルアーなど）を試す。← S03:p.61 / S05-A

第3章まとめ← S03:p.62

  • 活動のレベルを改善していくためにPDCAのサイクルを回そう
  • ゴールを意識して、PDCAを繰り返すことによって、計画の質を上げていこう（スパイラルアップ）
第4章

変更への対応力を磨く

早いアクションが怪我を浅くする

プロジェクトを実行していく中で、どこかで変更をせざるを得なくなる場合がある。その際に重要なのは対応の早さである。← S03:p.65

🎙 音声補足
旅行の例：計画時の変更はチケット払い戻し手数料程度だが、現地に着いてからの変更は全額支払い済みで払い戻し不可のケースもある。
システム開発の例：企画段階の変更は軽微なコストだが、プログラムを組んでしまってからの変更は大きなコストと時間がかかる。← S03:p.66-67 / S05-A

途中での変更をどう判断するか

プロジェクト中に目標に変更を与えるような事態が発生した場合の選択肢：← S03:p.68

  • A. すぐに目標を変更する
  • B. 一度決めたことなので元の目標で最後までやる
  • C. 目標を変更する必要があるのか、じっくりと検討し、必要があれば目標と同時に計画も変更する ← 推奨
🎙 音声補足
Aは「これまでかかったプロジェクトコスト」があるためすぐには選びにくい。Bは「誰も使わないシステムが出来上がる」可能性があり受け入れられない。仕事における最悪の対応は「結論を引き延ばすこと」。結論を引き延ばしている間プロジェクトは進行し悪い方向へ進んでしまう。← S05-A / S03:p.70

変更を受け入れて管理する

物事は変化するのが世の常。変化に柔軟に対応する一方で、目的（ゴール）は達成しなければならない。そこに至るアプローチ（進め方）は状況に応じて変化させてよい。← S03:p.71

🎙 音声補足 — 旅行の例
ゴールが「リゾートでゆっくりしたい」であれば、沖縄でなくても・飛行機でなくても近場でもよい。アプローチにこだわって「絶対飛行機で沖縄のこの日に行くんだ」と固執すると、空港で一夜を過ごして動けなくなるという悪い結果につながる。← S05-A / S03:p.71

変更の決定と連絡における重要事項：← S03:p.72

  • 計画の変更は公式な会議体などで決定する（一部のメンバーだけで決めない）
  • プロジェクトメンバー全員に変更内容を連絡し、新しい計画を伝達する
  • 変更連絡書：いつ・誰が発行したか・変更テーマ・変更内容・変更理由を周知する
🎙 音声補足 — よくある失敗例
変更があったのを知らずに、古い計画のままで一部のグループが進めていて全然合わなくなってしまうという例がよくある。変更内容の周知と新計画の徹底伝達が重要。← S05-A / S03:p.72

ゆとりの法則（トム・デマルコ氏）

【定義】ゆとりの法則

プロジェクトマネジメントの専門家トム・デマルコ（Tom DeMarco）氏が提唱。ゆとり（あそび）がないと実際の仕事はまわらないという法則。3×3マスのパズルにたとえると、1マス空いているから数字を動かせる。全部埋まっていたら何も動かせない。← S03:p.75

🎙 音声補足
スケジュールが全てのコマ数をギチギチに詰めたものだと、何か変更があったときに全く対処できず誰も身動きできなくなる。プロジェクト計画には必ずゆとりを作っておくことが重要なポイント。変更提案に柔軟に対応させるためにもある程度スケジュールにゆとりが必要。← S05-A / S03:p.75
鉄則：「計画は悲観的に、実行は楽観的に」
計画はどんなことが起こっても大丈夫な内容にする。実行は計測しながら楽観的に進める。
ただし予想外は必ず起きるので計画に時間をかけすぎてもダメ。
納得のいくレベルでの計画とその変更管理が成功のポイント。

📚 補足B: 知っておくべき背景知識
  • PDCAの起源と正確な歴史：PDCAの原型は1930年代に米国の統計学者ウォルター・シューハート（Walter A. Shewhart, 1891–1967）が品質管理のためのサイクルとして考案した「シュワートサイクル」に遡る。その後、弟子のW・エドワーズ・デミング（W. Edwards Deming）が1950年に日本科学技術連盟（日科技連）で講演した際にこのサイクルを紹介。この講演を聞いた日科技連の幹部が「Plan–Do–Check–Action」の形に整理して提唱したとされる（PDCAは日本発ともいえる）。デミング自身は晩年の1993年に「Check（確認）」を「Study（学習）」に改めた「PDSAサイクル」こそが正しいと主張し、PDCAは自分のモデルとは別物と証言している。日本ではQCサークル活動を通じて製造業に定着し、現在はISO 9001など国際規格にも組み込まれた。
    [外部補足 — Wikipedia「PDCAサイクル」/ iFinance「デミングサイクル」/ Asana「PDCAサイクル完全ガイド」各記事を照合済み]
  • KJ法の川喜田二郎氏（1920年5月11日 三重県生 – 2009年7月8日没、享年89歳）：地理学者・文化人類学者。京都帝国大学地理学科卒。東京工業大学名誉教授・筑波大学教授などを歴任。ネパール・チベットへのフィールドワークで蓄積した膨大なデータを整理する手法としてKJ法を1967年に開発・発表。著書『発想法 ―創造性開発のために』（中公新書、1966年刊）で手法を体系化。1984年にはアジアのノーベル賞ともいわれるマグサイサイ賞を受賞。KJ法は商標登録（株式会社川喜田研究所）されており、現在もビジネス・教育・研究の場で広く活用されている。
    [外部補足 — Wikipedia「川喜田二郎」/ コトバンク / イミダス各記事を照合済み]
  • トム・デマルコ（Tom DeMarco、1940年生）：米国のソフトウェアエンジニア・コンサルタント・著作家。コーネル大学BSEE・コロンビア大学修士・パリ大学ソルボンヌ校ディプロマを取得。マイクロソフト・アップル・HP・IBMなどのコンサルタントを務め、ニューヨーク/ロンドンを拠点とするAtlantic Systems Guild社の共同創設者。1999年スティーブン賞（ソフトウェア工学への貢献）受賞。
    主著として「ピープルウェア（Peopleware）」「デッドライン（The Deadline）」など。『ゆとりの法則（Slack: Getting Past Burnout, Busywork, and the Myth of Total Efficiency）』（2001年）では、ギチギチに詰まったスケジュールは変化への対応力を失わせるとして「余裕（Slack）こそが組織が変化に適応するカギ」と論じた。
    [外部補足 — Amazon「ゆとりの法則」書誌情報 / HackMD記事 / 日経BOOKプラス 各情報を照合済み]
  • OODAループの起源（ジョン・ボイド、John Boyd、1927–1997）：米空軍大佐・戦闘機パイロット・軍事戦略家。朝鮮戦争においてF-86でMiG-15と戦った経験から、機体性能で劣るF-86がなぜ圧倒的勝率（キルレシオ約10:1とも）を記録したかを分析。その要因を「パイロットの意思決定サイクルの速さ」に求め、Observe（観察）→ Orient（方向付け・状況判断）→ Decide（意思決定）→ Act（行動）のループを理論化した。著書は残さず、「Patterns of Conflict」などのブリーフィング資料として普及。退役後も国防総省のコンサルタントとして活動し、湾岸戦争（1991年）の作戦立案にも影響を与えたとされる。現在はビジネス・スポーツ・政策など幅広い分野に適用されている。
    [外部補足 — Wikipedia「OODAループ」「ジョン・ボイド（軍人）」/ coljohnboyd.com / Farnam Street記事 各情報を照合済み]
🎯 補足C: 試験・レポートで重要な観点
  • PDCAの各ステップの役割を説明できること：特にCheckとActionの違い（Checkは差異分析、Actionは次への改善計画立案）が問われやすい
  • 「なぜを5回繰り返す」の目的と意義：表面的対処と根本原因への対処の違い、5回繰り返すことで「プロジェクト内プログラミングルール整備」という本質的解決策にたどり着く例
  • KJ法の4ステップ：①要素を書く→②グループ化→③ラベル付け→④文章化、の順序と目的を説明できること
  • ゴール設定の5条件：全5条件を列挙し、「具体的であること（数値化）」がなぜ重要かを説明できること
  • PDCAとOODAの使い分け：PDCAは経験値が活きる繰り返し作業向け・OODAは未知の変化が激しい状況向け。どちらもループの反復が重要
  • 変更管理の3原則：①早期対応（コストを最小化）②公式会議体で決定③全メンバーへの周知徹底
  • 「計画は悲観的に、実行は楽観的に」の意味を説明できること
🔗 補足D: 関連概念・次回への接続
  • 第4回以降との接続：第3回ではPDCAの枠組みを学んだ。次回以降は「どのような計画（プラン）を立てるか」の詳細（WBS・スコープ管理・スケジューリングなど）に進む予定。[外部補足]
  • 第8回：品質マネジメント（QCの七つ道具）：第2章で触れたQCの七つ道具（パレート図・特性要因図など）の詳細は第8回で学ぶ。← S03:p.35
  • なぜを5回とトヨタ生産方式（TPS）：「なぜを5回繰り返す」はトヨタ生産方式（Toyota Production System）の要諦の一つ。TPSでは「問題の顕在化→真因追求→対策→標準化」のサイクルが重要視されており、「なぜを5回」は真因追求フェーズの具体的手法に当たる。トヨタでは「なぜなぜ分析」とも呼ばれる。[外部補足]
  • PDSAとPDCAの違い（デミングの主張）：デミング（1993年没年）はPDCAの「C（Check）」を「hold back（食い止める）」の意味にとられるとして好まず、「S（Study：学習・調査）」に置き換えた「PDSAサイクル」を主張した。PDCAは日科技連の幹部が整理した形であり、デミング自身はPDSAが本来の考え方に近いと証言している。[外部補足 — Wikipedia「PDCAサイクル」を照合済み]
  • アジャイル開発・OODAとの関係：PDCAのスパイラルアップはアジャイル開発（スプリント単位の繰り返し改善）と共通する考え方。OODAはPDCAが経験値を前提とするのに対し、未知の状況での即時判断を重視する点で補完的な関係にある。なお、Wikipediaによれば、トヨタ自動車はPDCAを重視しているが、実質的な運用はOODAループに近いという指摘もある。[外部補足]
  • SMARTゴールとの比較：本講義のゴール設定5条件は、経営・PM分野でよく使われるSMART基準（Specific・Measurable・Achievable・Relevant・Time-bound）と類似した構造を持つ。特に②具体的（S・M）、③期間決定（T）、①真の課題（R）が対応する。[外部補足]
